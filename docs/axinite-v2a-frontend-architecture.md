# Axinite v2a front-end architecture

## Front matter

- **Status:** Current design reference for the new SolidJS-based Axinite front
  end in this repository.
- **Scope:** The standalone front-end application under `axinite/`, the Bun
  mock backend and preview transport that support it locally, and the main
  architectural decisions that distinguish this implementation from the older
  browser gateway in `../axinite`.
- **Primary audience:** Maintainers and contributors who need to understand why
  this front end is structured as a separate application, how its runtime is
  split across client and preview layers, and which constraints should guide
  future changes.
- **Relationship to other documents:** `docs/v2a-front-end-stack.md` explains
  the broader v2a stack choices used across df12 mockups. This document explains
  how Axinite applies those ideas specifically, why some parts are already
  implemented here, and where this repo intentionally diverges from the older
  Axinite front end.

## 1. Purpose and architectural stance

This repository takes a different architectural approach from the older browser
surface in `../axinite`. The old front end is part of the Rust runtime. It is
served by the same binary that runs the agent, and its UI is generated by
checked-in `index.html`, `style.css`, and `app.js` assets embedded directly
into the host. By contrast, this repository treats the front end as its own
application runtime:

- the UI is a SolidJS single-page application (SPA),
- the browser code is built by Vite,
- the application state is expressed in TypeScript modules and components,
- the preview stack uses a Bun mock backend for realistic browser contracts, and
- the deployed artefact is a static build rather than a binary-embedded web
  shell.

That split is intentional. The goal here is not merely to redraw the existing
browser gateway. It is to establish a more explicit, testable, and maintainable
front-end architecture for Axinite while still preserving the productâ€™s current
information architecture, route structure, copy direction, semantic HTML, and
feature-flag behaviour.

In other words, the new front end is a standalone browser application that is
trying to stay truthful to Axiniteâ€™s runtime semantics, not a cosmetic mockup
that invents a separate product model.

## 2. The current system at a glance

Todayâ€™s implementation is composed of five layers:

1. **Build and packaging layer:** Bun runs scripts, Vite builds the SPA, and
   `vite-plugin-pwa` generates the service-worker-backed static artefact.
2. **Application shell layer:** SolidJS, TanStack Router, and a small provider
   tree define the shell, route tree, feature-flag context, and localisation
   context.
3. **Route and component layer:** route screens such as chat, memory, jobs,
   routines, extensions, and skills are implemented as focused Solid
   components, each backed by typed API modules.
4. **Preview integration layer:** TanStack Query, REST helpers, and
   `EventSource` clients connect the browser UI to the mock backend through
   JSON and Server-Sent Events (SSE).
5. **Local preview backend layer:** a Bun in-memory gateway mirrors the browser
   contract and is served behind the same origin as the built SPA during local
   preview runs.

This architecture keeps the browser concerns, preview transport, and runtime
simulation separate enough to reason about, while still feeling like one
coherent application during local development.

## 3. Build, delivery, and deployment model

The new front end is built from `axinite/` with Vite rather than being served
from embedded static files inside the host process. The main consequences are:

- `axinite/src/main.tsx` is the browser entry point and mounts the app after
  localisation initialization succeeds.
- `vite.config.ts` treats `axinite/` as the project root, emits the build into
  repository-level `dist/`, and configures the GitHub Pages base path.
- `scripts/postbuild-routes.mjs` mirrors route entry points into both root and
  deploy-prefixed paths so deep links work on Pages without server rewrites.
- `vite-plugin-pwa` generates the installable shell and service worker for the
  built preview artefact.

This is a significant shift from the old architecture in `../axinite`, where a
Rust rebuild is required for every browser change because the UI assets are
compiled into the host binary. The new approach accepts a more explicit front
end build step in exchange for a more normal SPA toolchain, faster iteration on
UI code, clearer asset boundaries, and better leverage of the TypeScript and
browser testing ecosystem.

## 4. Runtime composition inside the browser

### 4.1 Bootstrap and providers

The browser runtime is intentionally shallow:

- `axinite/src/main.tsx` waits for localisation readiness, then renders the app.
- `axinite/src/app/providers.tsx` wraps the tree in three providers only:
  TanStack Query, localisation, and feature flags.
- `axinite/src/app/router.tsx` defines the route tree and router base path.

That provider stack is a deliberate restraint. It keeps the global runtime
small and legible, and it avoids the â€śevery cross-cutting concern becomes a new
frameworkâ€ť failure mode that often makes front ends harder to evolve.

### 4.2 Routing

The route tree is explicit rather than generated. Each major top-level area is
defined in `axinite/src/app/router.tsx`, and `RoutePage` switches between the
route-specific preview components.

That choice was made for clarity, not because file-based generation is
inherently wrong. The route set is small and product-led. Keeping it explicit
makes it easier to see the public navigation surface, to keep feature flags
aligned with routes, and to preserve the route ordering expected by the shell.

### 4.3 State ownership

The front end uses a three-way state split:

- **TanStack Query** owns fetched runtime data and cache invalidation.
- **Solid signals and memos** own local interaction state, such as the active
  thread, composer draft, optimistic waiting state, and UI selections.
- **Context providers** own the small number of cross-cutting browser concerns,
  namely localisation and feature-flag resolution.

This is lighter than the fuller v2a stack described in
`docs/v2a-front-end-stack.md`. There is no Zustand store, no Dexie layer, and
no XState state machine in the current Axinite frontend. That is deliberate.
The present application needed a clear browser shell and realistic route flows,
not a maximal local-first platform on day one.

The teamâ€™s working assumption is that additional state machinery should be
earned by concrete complexity. Until that pressure appears, Query plus signals
plus narrow contexts remain easier to read, easier to test, and easier to keep
faithful to the product surface.

## 5. Data and integration architecture

### 5.1 Typed API modules instead of in-component fixtures

One of the most important architectural moves in this repo was replacing
route-local fixture arrays with typed API modules. The browser no longer treats
chat, memory, jobs, routines, extensions, skills, logs, gateway status, or
feature flags as local mock data hidden inside each screen. Instead:

- request and response shapes live in `axinite/src/lib/api/contracts.ts`,
- route-specific client helpers live in `axinite/src/lib/api/*.ts`, and
- components consume those helpers through Query and mutations.

This is not just a code organisation preference. It is what allows the UI to
act like a real application surface rather than a static storyboard. It also
means the browser contract is visible and testable in one place, instead of
being recreated ad hoc in several components.

### 5.2 REST and SSE, but not the old host-integrated transport mix

The current browser integration model is intentionally simpler than the old one
in `../axinite`:

- JSON over `fetch` is used for request-response interactions.
- SSE via `EventSource` is used for live chat and log updates.
- There is no browser-side WebSocket transport in this repoâ€™s preview stack.
- There is no authentication token flow in the local preview architecture.

The old browser gateway needs a broader transport surface because it is a real
hosted control plane inside the runtime. It must deal with authentication,
optional query-string token fallbacks for streaming, and a larger set of
runtime-backed channel behaviours. The new front end narrows that surface on
purpose. It focuses on the browser interaction model that the UI actually
needs, while leaving production auth and host integration concerns to the real
runtime in `../axinite`.

### 5.3 Same-origin preview transport

The preview architecture makes a strong commitment to same-origin delivery
during local development:

- `scripts/dev.ts` starts the mock API, build watcher, and preview server
  together.
- `mock-backend/src/preview-server.ts` serves built assets from `dist/`.
- `/api/*` is proxied through the same browser origin to the mock API.

This was not accidental. An earlier `http-server` approach proved too brittle
for the required path-preserving proxy behaviour. The current Bun preview
gateway was chosen because the browser code assumes relative `/api/*` requests,
and the team wanted to preserve that contract instead of introducing a
preview-only API base URL.

### 5.4 The mock backend is a contract surface, not a fake product

The Bun mock backend is intentionally an in-memory potemkin village. It is rich
enough to prove the browser contract, but deliberately stops short of becoming
a second implementation of the Axinite runtime.

That means it does provide:

- realistic route payloads,
- mutable request flows,
- chat and log SSE streams,
- route-level summaries and detail panes, and
- enough behaviour to make optimistic UI, loading, and control actions feel
  real.

It does not attempt to provide:

- durable persistence,
- production authentication,
- deep runtime orchestration,
- host-managed session semantics identical to `../axinite`, or
- a byte-for-byte simulation of the Rust application.

This boundary matters. Without it, the preview stack would become expensive to
maintain and would drift into building a parallel backend product.

## 6. User interface architecture

### 6.1 Shell and navigation

The shell is implemented in `axinite/src/components/app-shell.tsx`. It owns:

- the top bar and route navigation,
- locale selection,
- the logs surface,
- the gateway status badge, and
- the debug-flag panel when enabled.

The shell is intentionally persistent across route changes. That keeps the
Axinite experience closer to an application workspace than to a series of
isolated pages.

### 6.2 Route previews as focused domain surfaces

Each major route is implemented as a dedicated preview component, for example
`ChatPreview`, `MemoryPreview`, `JobsPreview`, `RoutinesPreview`,
`ExtensionsPreview`, and `SkillsPreview`.

This is a more component-oriented approach than the old front endâ€™s large
hand-written `app.js`, where one runtime script owns rendering and behaviour
for the whole browser experience. The new structure improves locality:

- route-specific behaviour stays with the route component,
- shared shell concerns stay in the shell,
- API concerns stay in `lib/api`, and
- semantic styling stays in the CSS layer.

### 6.3 Semantic CSS over utility sprawl

Although the stack uses Tailwind and DaisyUI in the build, the Axinite frontend
leans heavily on semantic classes in `axinite/src/styles/semantic.css`.

That was a conscious choice. This repo values semantic HTML, stable class
names, and design clarity over a utility-only approach. The CSS therefore acts
as a product-facing design language, not just a pile of layout tokens attached
to individual elements.

This is another divergence from the old front end. The older gateway UI is
hand-authored CSS coupled directly to static HTML and DOM-generation code. The
new frontend keeps semantic styling, but moves it into a component-oriented SPA
layout where routes and styles can evolve together without a monolithic script
owning all rendering concerns.

## 7. Localisation and feature-flag decisions

### 7.1 Localisation

The front end uses `i18next` with Fluent bundles, a narrow detection order, and
document-level `lang` and `dir` updates. A key design choice here is that the
runtime only advertises fully translated locales through `AVAILABLE_LOCALES`.

That means localisation is treated as a product-integrity concern, not a
cosmetic toggle. The app would rather expose fewer languages than offer a mixed
language shell.

### 7.2 Feature flags

Feature flags are resolved by merging three layers:

1. registry defaults,
2. backend-provided values from `/api/features`, and
3. local overrides stored in browser storage.

This design gives the UI a realistic runtime-controlled route surface while
still preserving local debugging controls. The decision was important because
the prototype needs to behave like a browser client under runtime flags, not
like a purely static showcase.

## 8. Testing and verification approach

The new frontend is built to be verifiable in layers:

- unit and behaviour tests use Vitest with Happy DOM,
- accessibility tests run separately through an a11y-specific Vitest config,
- end-to-end checks run through Playwright against the built preview stack, and
- the repoâ€™s formatting, linting, typechecking, markdown, and diagram gates are
  all treated as normal part of delivery.

This is materially different from the old front end, where the browser surface
is tightly coupled to the Rust host and its validation story is therefore more
host-centric. The new architecture makes the browser layer a first-class target
for front-end-focused tests, without requiring the full Rust runtime to be
booted for every UI change.

## 9. How this differs from the older front end in `../axinite`

The differences can be summarised as follows.

### 9.1 Deployment boundary

- **Old front end:** embedded in the Rust binary and served by the host.
- **New front end:** built as a standalone SPA and served as static output in
  preview.

### 9.2 Rendering model

- **Old front end:** hand-written DOM construction in one large browser script.
- **New front end:** component-based SolidJS rendering with explicit route and
  provider composition.

### 9.3 Runtime coupling

- **Old front end:** directly coupled to real runtime subsystems, auth, and
  gateway middleware.
- **New front end:** coupled to typed browser contracts and a preview gateway,
  not to the host process itself.

### 9.4 Transport model

- **Old front end:** broader runtime-facing mix of static hosting, authenticated
  APIs, SSE, and WebSocket support.
- **New front end:** narrower browser contract centred on JSON plus SSE, behind
  a same-origin preview gateway.

### 9.5 State and code organisation

- **Old front end:** one large script orchestrates most browser state and DOM
  updates.
- **New front end:** route components, API modules, provider contexts, and
  semantic CSS divide responsibilities more cleanly.

### 9.6 Product intent

- **Old front end:** an operational browser surface for the running agent.
- **New front end:** a new front-end architecture and preview stack intended to
  validate the future browser experience before it is folded back into the
  broader Axinite product story.

## 10. Main decisions and considerations

The main architectural decisions in this repo were:

- **Keep the browser contract realistic, even in preview.** That is why the
  mock backend exists and why route components do not rely on hidden fixture
  arrays anymore.
- **Prefer a separate SPA build to embedded assets.** The team wanted a front
  end that can use modern browser tooling, typed modules, component tests, and
  route-scoped UI architecture.
- **Stay same-origin during preview.** Relative `/api/*` calls are part of the
  product contract, so the preview stack was made to preserve that instead of
  teaching the browser a preview-only base URL.
- **Use the lightest state model that remains legible.** Query plus signals and
  context won over a larger state stack because the current surface does not yet
  justify the added indirection.
- **Protect semantic and localisation integrity.** Semantic CSS classes,
  semantic HTML, and complete-locale gating are treated as architectural
  concerns, not only implementation details.
- **Optimise for future integration, not for permanent mock isolation.** The
  mock backend is useful precisely because it preserves browser-facing
  contracts, making later integration with the fuller product easier.

The main constraints that continue to matter are:

- this repo is still a prototype-first frontend rather than the final deployed
  Axinite site,
- build and delivery must keep working under the GitHub Pages base path,
- feature flags and locale behaviour must stay truthful to runtime expectations,
  and
- new complexity should be added only when the browser experience clearly needs
  it.

## 11. Implications for future work

If this front end continues to mature, future changes should preserve the
architectural split established here:

- keep product-facing route contracts typed and explicit,
- keep preview transport same-origin unless there is a compelling reason to
  change the browser contract,
- treat localisation completeness and semantic accessibility as architecture,
  not polish,
- resist introducing broader state frameworks until there is concrete pressure,
  and
- use the old `../axinite` browser gateway as a behavioural reference, not as
  a structural template.

That last point is the most important. The older Axinite front end explains the
runtime semantics that the browser must honour. It should not automatically
dictate the implementation shape of the new frontend. The whole point of this
repo is to prove that Axinite can keep its browser-facing behaviour while
moving to a more maintainable, more testable, and more modern front-end
architecture.
